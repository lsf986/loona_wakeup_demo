# Loona 语音唤醒模块实现方案（v1.0）

> 关联需求：《语音交互需求-20260430.pdf》§7.1 语音唤醒、§8.1 无唤醒词直接承接、§8.2 唤醒后立即短句、§9 非功能需求、§10.1.1 唤醒验收、§11 风险点
> 文档定位：**工程可落地实现方案**，聚焦"桌面办公场景 + 真实环境噪声"下的唤醒链路设计、算法选型、参数、降级与验收。
> 关联模块：注意力模块、环境感知模块、状态管理模块、判停子模块（§7.2）

---

## 1. 目标与关键指标

### 1.1 功能目标
1. 持续监听（含 TTS 播报期间），识别有效唤醒意图后进入本轮语音承接；
2. 同时支持 **唤醒词（KWS）** 与 **无唤醒词多模态意图**（§7.1.2-1、§8.1）两条触发路径；
3. 唤醒→承接之间音频不丢、首句不吞（§7.1.3、§9.1-2）；
4. 连续/重复唤醒不混乱（§7.1.2-4）；
5. 在桌面环境噪声下控制误唤醒、控制漏唤醒。

### 1.2 关键指标（验收口径）

| 指标 | 目标值 | 测量场景 |
|---|---|---|
| 唤醒词 FRR（漏唤醒率） | ≤ 3 % | 安静 + 40 dB 粉噪，1 m 正对 |
| 唤醒词 FAR（误唤醒率） | ≤ 0.5 次 / 24 h | 连续播放办公室真实录音 + 视频 BGM |
| 无唤醒词路径误进入率 | ≤ 1 次 / 8 h | 有人在场但未对 Loona 说话 |
| 端到端唤醒时延（VAD起点→进入监听） | ≤ 300 ms | 设备侧计时 |
| 唤醒→承接音频连续性 | 首句前 500 ms 音频 0 截断 | 环形缓冲回灌验证 |
| TTS 自唤醒率 | 0（硬约束） | 回环播放 Loona 自身 TTS 语料 |
| 短句承接成功率（≤2 字） | ≥ 95 % | §8.2 场景 |

## 2. 总体架构

```
 麦克风阵列 ──► 前端信号处理 ──► 环形缓冲(Ring Buffer, 2s)
                   │                 │
                   ▼                 ▼
           ┌─────────────┐   ┌─────────────────┐
           │ AEC/NS/AGC  │   │  VAD (轻量)     │
           │ DOA/BF      │   └──────┬──────────┘
           └──────┬──────┘          │
                  │ 主通道 + 参考通道│
                  ▼                 ▼
          ┌──────────────────────────────────┐
          │  KWS 唤醒词引擎（Always-on）     │
          │  ——两级：Tiny Spotter → Verifier │
          └──────┬───────────────────────────┘
                 │ 命中/候选
                 ▼
          ┌──────────────────────────────────┐
          │  唤醒仲裁器 (Wake Arbiter)       │
          │  融合：KWS 分数 + VAD + DOA +    │
          │        注意力(人脸/唇动/眼神) +  │
          │        环境感知(TTS播报/声压级)   │
          └──────┬───────────────────────────┘
                 │ 有效唤醒事件
                 ▼
          ┌──────────────────────────────────┐
          │  语音承接切换器 (Handoff)         │
          │  — 环形缓冲回灌 + 状态机跳转      │
          └──────────────────────────────────┘
```

**关键设计原则：**
- **始终录音，事件判决**：音频链路 24 h 不关，所有判决用环形缓冲的历史数据做，避免切换丢帧（解决 §7.1.3 "音频流截断"）。
- **分层决策**：Tiny 模型跑在 DSP/低功耗核心，Verifier 跑在 AP，仲裁器跑在业务层——兼顾功耗与精度。
- **多模态是加权，不是串联**：单一模态不足以判决，但任一模态也可作为强证据触发（见 §5 仲裁规则）。

## 3. 前端信号处理（噪声鲁棒性的核心）

桌面办公场景下的主要噪声：

| 噪声类型 | 特征 | 对策 |
|---|---|---|
| 键盘/鼠标敲击 | 瞬态、宽频 | NS + 瞬态抑制（transient suppressor） |
| 风扇/空调/主机噪音 | 稳态、低频 | 谱减法 NS + 高通滤波（80 Hz） |
| 同事远场交谈 | 远场、DOA 不在正前 | 固定波束 + DOA 距离门限 |
| 电视/视频 BGM、人声 | 已知频谱（若可获取）或远场 | AEC（若同一设备播放）+ 环境感知声纹黑名单 |
| Loona 自身 TTS 回放 | 参考信号可得 | **线性 AEC + 残差 NLP**（硬性去除） |
| 多人同时说话 | 混叠 | BF + 主 DOA 跟踪 |

### 3.1 推荐方案
- **麦克风阵列**：≥ 2 mic（线阵或小圆阵），用于 DOA 估计与固定/自适应波束形成；若硬件为单 mic，跳过 BF，DOA 置信度置 0。
- **AEC**：基于 SpeexDSP / WebRTC APM（APM 3A）或自研 FDAF + NLP 双级；**参考信号必须取扬声器功放前数字信号**（不是播放文件），否则非线性残留会造成自唤醒。
- **NS**：WebRTC NS + 瞬态抑制；aggressiveness = 2（平衡漏识别与去噪）。
- **AGC**：目标电平 -18 dBFS，限幅 -3 dBFS，防止 KWS 在用户远讲/轻声时输入过小。
- **VAD**：WebRTC VAD mode=2 或 Silero VAD（更鲁棒，量化后 < 1 MB），用于门控 KWS（节流）和给仲裁器提供 "当前是否语音" 证据。

### 3.2 音频流与缓冲
- 采样率 16 kHz / 16-bit / 单通道（送后级）；
- **环形缓冲 2 s**（含处理后音频），供唤醒命中后回灌，保证首句不丢（§7.1.2-2、§9.1-2）；
- 回灌起点：`max(唤醒事件时间 - 1.5s, VAD 上升沿 - 200ms)`，取二者较早，既捕捉无唤醒词场景下的完整语音，又避免带入过多前导噪声。

## 4. KWS 唤醒词引擎

### 4.1 两级结构

| 级别 | 部署 | 模型 | 作用 | 触发率 |
|---|---|---|---|---|
| L1 Tiny Spotter | DSP / 低功耗核 | CRNN / DS-CNN，≤ 100 kB，≤ 3 MFLOPs/帧 | 粗筛，24 h 常开 | 宽松阈值，保召回 |
| L2 Verifier | AP | 更大 CRNN / Transformer，5–20 MB，带说话人嵌入 | 二次验证 + 置信度打分 | 只在 L1 命中后跑 |

### 4.2 训练与数据
- **正样本**：唤醒词录制 ≥ 10 k 条，覆盖男女/方言口音/远近场/不同音量；
- **难负样本挖掘**：
  - 唤醒词的音素近邻（同韵尾/同声母的常用词）；
  - **影视剧 + 办公室环境长音频** 连续回放挖掘 FP；
  - Loona 自身 TTS 语料（含所有系统提示语）——**强制 0 自唤醒**；
- **数据增强**：RIR 卷积（桌面 0.5–2 m）、加噪（MUSAN，SNR 0–20 dB）、时域速度扰动 ±10 %、随机增益 ±10 dB。

### 4.3 阈值策略
- L1 阈值：保证召回 ≥ 99 %，允许 FAR 偏高；
- L2 阈值：**动态阈值**，基于环境 SNR 调整：
  - SNR ≥ 20 dB：阈值 0.5（宽松）
  - 10 ≤ SNR < 20：0.65
  - SNR < 10：0.8（严格，避免噪声触发）
  - TTS 播报期间：+0.1（更严，即使 AEC 生效也加保险）。
- **自适应冷却**：5 s 内连续命中 ≥ 3 次 → 提高阈值 0.1 并上报"疑似误唤醒环境"，5 min 后恢复（对应 §7.1.2-4 "连续多次触发不混乱"）。

## 5. 唤醒仲裁器（核心去噪/防误触设计）

仲裁器是"结合多模态 + 环境 + 上下文"做最终判决的位置（§7.1.2-1、§7.1.2-5）。采用**证据加权评分 + 硬门限**：

### 5.1 证据源
| 证据 | 来源 | 产出 | 典型权重 |
|---|---|---|---|
| E1 KWS 分数 | L2 Verifier | [0,1] | 1.0 |
| E2 DOA/距离 | 阵列 + 环境感知 | 在 ±60° 且距离 ≤ 1.5 m → 1，否则衰减 | 0.6 |
| E3 人脸朝向/眼神 | 注意力模块 | 正面/注视 → 1；偏头 → 0.3；无人 → 0 | 0.7 |
| E4 唇动 | 注意力模块 | 命中时段唇动匹配 → 1 | 0.5 |
| E5 声纹（可选） | 本地 speaker embedding | 与已注册主用户相似度 | 0.4（辅助） |
| E6 TTS 自播报 | 环境感知 | 正在播报 → 证据强衰减（×0.3） | 负向 |
| E7 短时背景稳定性 | 前端 | 触发瞬间是否存在长时段稳定人声/BGM | 负向 |

### 5.2 两条触发路径

**路径 A：唤醒词路径**
```
if L2_score ≥ 动态阈值:
    score = E1 × (0.6 + 0.2·E2 + 0.2·E3)   # DOA/眼神作为加分项，不必需
    score -= 0.3·E7 + 0.4·(TTS播报中且AEC残差能量>阈值)
    if score ≥ 0.6:  trigger
```
唤醒词路径**不强依赖**视觉证据，保证黑暗/背对/无人脸检测时仍可唤醒（可用性）。

**路径 B：无唤醒词多模态意图路径**（§8.1）
```
需同时满足（硬门限，避免误进入）：
  1) 注视 Loona 持续 ≥ 600 ms（E3≥0.8）
  2) 检测到明确唇动 且 VAD 上升沿 ≤ 300 ms 内对齐（E4≥0.8）
  3) DOA 与人脸方位一致（|Δ| ≤ 20°）
  4) 非 TTS 播报中，或播报已淡出 > 500 ms
  5) 近场（距离 ≤ 1.2 m）
满足后进一步打分：
  score = 0.4·E3 + 0.3·E4 + 0.2·E2 + 0.1·E5
  if score ≥ 0.75:  trigger
```
该路径**严格**：宁可漏，不可错（§11.1-1 "多模态意图判断过于激进会带来更多误触发"）。

### 5.3 负向强约束（硬性一票否决）
- TTS 正在播报 且 AEC 后残差能量 / 参考信号能量 > 0.2 → 直接丢弃唤醒候选（极大概率是回环自唤醒）；
- 触发时刻前 2 s 音频中 VAD 连续占比 > 80 %（典型电视剧/长对话背景） → 需 E3 或 E4 至少一项为强证据，否则丢弃；
- 静音环境（整体 RMS < -55 dBFS）下 KWS 命中 → 视为异常，丢弃并上报。

## 6. 承接切换与首句保护（§7.1.2-2, §7.1.3, §9.1-2）

1. 唤醒事件产生后，**并行**执行：
   - a. 状态机切到 `LISTENING`；
   - b. 从环形缓冲回灌音频至 ASR 前置队列（起点见 §3.2）；
   - c. 触发反馈（见 §7）。
2. ASR / 判停模块从回灌起点开始连续处理；唤醒词本身的音素段**按是否属于用户表达** 决定截除：
   - 唤醒词路径：截除唤醒词对应时间段（KWS 返回端点）后仍保留 100 ms 前导与 100 ms 后导缓冲；
   - 无唤醒词路径：不截除，全部送入。
3. **短句场景保护**（§8.2）：
   - 判停模块收到 `HANDOFF_SHORT_UTTERANCE_HINT`（当 KWS 端点后 ≤ 200 ms 内仍有 VAD 活动时置位），将判停尾端静音阈值从 600 ms 收紧为 400 ms，同时最小语音长度下限从 300 ms 降到 120 ms，避免误判为误唤醒。

## 7. 唤醒反馈（§7.1.2-3）

唤醒确认必须 **≤ 150 ms** 内给出可感知反馈，以降低用户重复呼叫概率：

| 反馈 | 实现 | 时机 |
|---|---|---|
| 表情 | 屏幕表情切"在听"态 | 触发同步 |
| 动作 | 头部轻微朝向 DOA | 触发后 80–200 ms |
| 状态指示 | LED / UI 状态环 | 触发同步 |
| 简短语音（可关） | "嗯？"/"在"，低音量 TTS | 仅在唤醒词路径且用户开启时 |

反馈产生的音频必须接入 AEC 参考信号，避免自唤醒。

## 8. 状态机

```
           ┌───────────┐
           │  IDLE     │ (持续监听，KWS/多模态运行)
           └─────┬─────┘
                 │ Wake event
                 ▼
           ┌───────────┐
           │ LISTENING │──判停─► PROCESSING ──► TTS/回复
           └─────┬─────┘
                 │ 超时 8s 无有效语音
                 ▼
             回到 IDLE 并上报 "唤不起正文"
```

- 连续唤醒：在 LISTENING/PROCESSING 期间再次收到唤醒事件 → 视为**用户打断**，交给 §7.5 打断模块；**不重复触发新一轮唤醒反馈**（§7.1.2-4）。
- 所有状态跳转需写入埋点日志（§10）。

## 9. 异常与降级

| 场景 | 处理 |
|---|---|
| AEC 回声消除模块崩溃 | 自动禁用无唤醒词路径，唤醒词路径阈值 +0.15；上报 |
| 视觉模块（注意力）离线 | 无唤醒词路径临时关闭，唤醒词路径正常（保证基础可用） |
| 连续 5 min 检测到高 FAR 环境 | 动态抬高阈值并通知用户（UI 提示"环境嘈杂，已启用严格唤醒"） |
| KWS 模型加载失败 | 回退到仅多模态路径 + 物理按键唤醒兜底 |
| 麦克风权限/硬件异常 | 进入"不可唤醒"态，UI 明示，引导排查 |

## 10. 埋点与离线闭环

每次唤醒事件上报（脱敏后，用户可关）：
- 时间戳、路径类型（KWS/多模态）、L1/L2 分数、各证据值、SNR 估计、DOA、是否 TTS 中、最终判决；
- 命中前后各 2 s 音频（仅在用户授权的"体验改进计划"开启时）；
- 用户在 5 s 内是否产生有效 ASR 文本 → 反推该次唤醒是否为真唤醒（正/误样本自动标注）。

用于：
- 持续迭代 KWS 负样本库；
- 仲裁器权重与阈值的灰度 A/B；
- 建立"按用户/按环境"的个性化阈值（长期）。

## 12. 里程碑建议（实现顺序）

1. **M1 基础链路打通**：单 mic + WebRTC APM + 单级 KWS + 环形缓冲回灌；跑通承接，满足 §10.1.1 基础验收。
2. **M2 噪声鲁棒性**：加入阵列 / DOA / AEC 参考信号闭环 / 难负样本训练；达到 FAR / 自唤醒指标。
3. **M3 多模态路径**：接入注意力模块证据，开启路径 B 的灰度。
4. **M4 个性化与在线闭环**：埋点回流、动态阈值、主用户声纹（可选）。

每个里程碑必须跑 §1.2 指标回归测试集才能进入下一阶段。
